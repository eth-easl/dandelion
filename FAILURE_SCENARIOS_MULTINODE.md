## Failure Matrix


| Scenario                                                                    | What happens in Dandelion                                               | Covered before these changes? | Covered by current multi-node changes? | Notes                                                                                     |
| --------------------------------------------------------------------------- | ----------------------------------------------------------------------- | ----------------------------- | -------------------------------------- | ----------------------------------------------------------------------------------------- |
| Remote node crashes before accepting offloaded work                         | Work may need to remain local or be retried elsewhere                   | Partially                     | No                                     | Depends on existing queue/offload behavior; no durable multinode handoff protocol         |
| Remote node crashes after accepting work but before producing a response    | Work may be lost or may be re-executed                                  | No                            | No                                     | No durable cross-node execution record or ownership transfer                              |
| Remote node completes work but crashes before sending the response          | Owner may not know whether work finished                                | No                            | No                                     | Classic uncertain-completion window                                                       |
| Socket fails before work handoff is complete                                | Owner may not know whether the remote ever received the work            | No                            | No                                     | A relay or MITM can simulate this ambiguous handoff boundary                              |
| Socket fails while remote work is executing                                 | Owner may lose visibility into in-flight remote execution               | No                            | No                                     | Owner cannot distinguish remote crash from network failure from delayed completion        |
| Socket fails after remote completion but before response delivery           | Owner may retry even though remote work already finished                | No                            | No                                     | Completed-but-undelivered is indistinguishable from unfinished without a durable protocol |
| Socket delivers truncated, malformed, or dropped messages                   | Invocation or response state may become ambiguous                       | No                            | No                                     | Current socket handling is not designed as a resilient message protocol                   |
| Network partition between owner and remote node during execution            | Owner may time out while remote continues or finishes                   | No                            | No                                     | Can lead to duplicate execution if owner retries elsewhere                                |
| Offload request is delivered more than once                                 | Same remote work may execute multiple times                             | No                            | No                                     | No dedupe or exactly-once offload protocol                                                |
| Response from remote node is delivered more than once                       | Owner may observe duplicate completions                                 | Partially                     | No                                     | Requires idempotent response handling to be fully safe                                    |
| Owner node crashes while remote work is still running                       | Remote work may become orphaned                                         | No                            | No                                     | No recovery protocol yet for reclaiming or cancelling orphaned remote work                |
| Owner node crashes after remote result arrives but before owner persists it | Result may be lost from the owner perspective                           | No                            | No                                     | Single-node recovery does not yet extend to remote execution state                        |
| Remote node crashes after causing external side effects                     | Side effects may be replayed if the owner resubmits                     | No                            | No                                     | Same at-least-once problem, now with an extra network boundary                            |
| Remote data reference is lost or remote data server becomes unavailable     | Downstream nodes may fail to resolve required data                      | Partially                     | No                                     | Current remote-data path is availability-sensitive                                        |
| Remote data is deleted too early                                            | A consumer may fail while resolving a still-needed remote reference     | Partially                     | No                                     | Lifetime coordination for remote data is inherently distributed                           |
| Remote data delete message is lost                                          | Data may leak remotely                                                  | Partially                     | No                                     | Resource leak rather than direct correctness failure                                      |
| Dynamic registry state differs across nodes                                 | Remote node may not understand or may execute a different function      | No                            | No                                     | No cluster-wide registry consistency mechanism                                            |
| Composition metadata differs across nodes                                   | Remote execution semantics may diverge                                  | No                            | No                                     | Stable node IDs on one node do not imply cluster-wide consistency                         |
| Binary/function implementation differs across nodes                         | Same function id may execute different code on different nodes          | No                            | No                                     | Single-node binary hash validation does not solve cross-node consistency                  |
| Invocation ownership is ambiguous after retries or failover                 | Multiple nodes may believe they are responsible for the same invocation | No                            | No                                     | Would need leasing, fencing, or a durable owner election model                            |
| Node membership or capacity information is stale                            | Work may be offloaded to nodes that are unavailable or overloaded       | Partially                     | No                                     | A scheduling/control-plane robustness issue                                               |
